# TradeLife v3 — Full System Audit

## Audit Scope
- Source of truth audited: codebase in `/app`
- Audit method: schema migrations, server actions, API routes, UI routes, supporting workers, and integration wiring
- Comparison baseline: TradeLife OLD blueprint + TradeLife v3 MASTER blueprint
- Constraints honoured: code-first verification only; no assumptions from file names alone

## Observed Runtime Architecture
- Primary app: Next.js 14 App Router application in `/app/app`
- Data layer: Supabase Postgres + Storage + service-role usage across server actions and API routes
- “Backend” service in `/app/backend/server.py` is only a thin FastAPI proxy forwarding `/api/*` to the Next.js app; it is not the business-logic source of truth
- Business logic is split across:
  - Next.js route handlers in `/app/app/api/*`
  - Server actions in `/app/lib/actions/*`
  - SQL migrations in `/app/supabase/migrations/*`
- Current codebase is still architected as **TradeLife v2 with partial v3 overlays**, not a clean v3 system

## 2) MODULE-BY-MODULE AUDIT

### MODULE: Multi-tenant architecture / org isolation / auth context

**STATUS:** ⚠️ Partial

**DETAIL:**
- What exists
  - `org_id` is present on most operational tables in the migrations.
  - Generic RLS patterns exist in `00001_foundation_schema.sql` and later migrations.
  - User/org bootstrap exists via `00002_auth_auto_seed.sql` and `lib/auth/getUser.ts`.
- What is missing
  - A consistent tenant-safe public access model.
  - A formal active-org switching model beyond `active_org_id` field presence.
  - End-to-end separation between authenticated routes and public token routes.
- What is incorrect
  - `quotes` has a public SELECT policy of `share_token IS NOT NULL` in `00001_foundation_schema.sql`. Because `share_token` is auto-generated by default, that effectively makes **all quotes publicly selectable**.
  - `portal_invites` has `CREATE POLICY "Public token verify" ... USING (true)` in `00016_client_portal.sql`, which exposes invite rows broadly instead of only per-token verification.
  - `/app/app/api/auth/me/route.ts` returns the **first available profile/org** when no session exists.
  - `/app/app/api/transactions/route.ts` and `/app/app/api/user-rules/route.ts` trust caller-supplied `org_id` while using service-role access.

**RISK:** High

- What is incorrect
  - The migration chain defines **two incompatible ledger schemas**:
    - `00004_job_wallets.sql` creates `job_wallet_ledger` with `wallet_id`
    - `00006_financial_engine.sql` creates `job_wallet_ledger` with `job_id`, `category`

  - Later migrations `00007`, `00008`, and `00010` assume `category` and `job_wallet_ledger_category_check` already exist, which is not true if `00004` ran first.

## 3) CRITICAL FAILURES

1. **Schema migration chain is broken / non-reproducible.**
   - `job_wallet_ledger` is defined incompatibly in `00004` and `00006`.
   - `quote_snapshots` is defined twice, with later code expecting columns the earlier table does not create.

2. **Public data leakage through RLS.**
   - All quotes are effectively public because of `share_token IS NOT NULL` policy.
   - Portal invite verification policy is globally open.

3. **Revenue recognition is financially incorrect.**
   - Revenue is recognised on invoice creation instead of payment receipt.
   - Payment posting then uses a different category (`REVENUE`) and double-counts.

4. **Stripe integration is not production-safe.**
   - No verified signature handling.
   - No webhook-driven payment lifecycle.
   - No refund or dispute handling.

5. **Public quote acceptance flow is broken.**
   - Frontend request payload and backend route contract do not match.
   - The route writes an invalid enum value (`accepted`).

6. **Core pages are placeholders.**
   - Quote create, calendar, assistant, finance, clients, jobs detail, and some API routes are placeholders or JSON dumps.

7. **Several routes target tables that do not exist in migrations.**
   - `proposals`, `transactions`, `user_rules`, `twilio_recordings`, `webhook_logs`.

8. **Unauthenticated user context leakage exists.**
   - `/api/auth/me` falls back to the first profile/org even without a session.

9. **Lineage is not enforced end-to-end.**
   - Jobs can be manually created without a source quote in UI code.

10. **The ledger is not a trustworthy single source of truth.**
    - Schema, categories, writers, and dashboards disagree with one another.

## 4) ARCHITECTURAL RISKS

- **Migration drift risk:** clean deploys can fail or create a schema the app does not actually expect.
- **Service-role sprawl:** business logic frequently bypasses RLS in route handlers and server actions.
- **Duplicate workflow implementations:** multiple send/accept/payment paths with different logic cause state divergence.
- **No transactional orchestration:** quote→job, invoice→ledger, and payment→ledger flows are not atomic.
- **Mock/live boundary confusion:** dashboard and AI interfaces visually imply live capability but are partly mocked.
- **Schema-contract drift:** routes, actions, and pages reference missing tables/columns/relations.
- **Security model inconsistency:** public token flows, service role use, and RLS policies are not designed as one coherent model.
- **Thin backend proxy adds complexity without control:** FastAPI proxy layer adds operational surface but not real domain logic.

## 8) NEXT 10 ACTIONS (ORDERED)

1. **Rebuild the database baseline**
   - Replace conflicting migrations with one canonical v3 schema or a verified squash migration.

2. **Lock down security immediately**
   - Remove global public quote / portal-invite policies and fix service-role endpoints.

3. **Define a canonical ledger contract**
   - Standardise columns, categories, direction rules, and immutability.

4. **Re-implement payment posting properly**
   - Quote accepted → `COMMITTED_REVENUE`
   - Payment received → `RECOGNISED_REVENUE`
   - Expense confirmed → `EXPENSE` + `VAT`
   - Refund → negative recognised revenue

5. **Build a real Stripe webhook processor**
   - Signature verification, event persistence, idempotency, invoice/payment updates, refund/dispute handling.

6. **Collapse duplicate quote workflows into one state machine**
   - One send path, one accept path, one quote→job conversion path.

7. **Build the actual quote creation/proposal flow**
   - Real create page, client/reference capture, full line items, PDF/proposal output, snapshot correctness.

8. **Fix job and invoice lineage**
   - No manual orphan job creation; invoice generation must copy the correct job scope and support partial payments.

9. **Implement the real scheduling system**
   - Booking states, conflict detection, locks, resource allocation, payment gating.

10. **Remove or replace placeholders with live modules**
    - Assistant, calendar, finance, clients, jobs detail, dashboard, integration routes.

## Bottom Line

TradeLife is **not a v3 system today**.

It is a partially upgraded v2 codebase with:
- meaningful domain intent,
- real schema and workflow effort,
- but severe structural drift between migrations, actions, routes, and blueprint expectations.

The biggest reality gap is this:

**The UI suggests a connected operating system. The underlying implementation is not yet coherent enough to safely run money, scheduling, or public client flows.**
